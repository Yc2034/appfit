# Image Deployment Suggestions

这个文档用于约束 `AppFitMVP` 的本地图片资源，目标是：

- 真机显示清晰
- 包体增长可控
- 后续补图时不需要反复猜尺寸

## 1. 当前页面类型

项目里目前主要有两类图片：

1. 课程封面
2. 动作图 / 动作示例图

它们在 UI 里的使用方式不一样，所以不要用同一套尺寸粗暴处理。

## 2. 推荐分辨率

### 2.1 课程封面

适用位置：
- Tab 1 课程卡片
- 课程详情页 Hero 封面

推荐原图：
- `1600 x 900`
- 比例：`16:9`

可接受下限：
- `1280 x 720`

不建议超过：
- `2000 x 1125`

原因：
- 当前课程卡片和详情 Hero 都是横向封面
- `16:9` 裁切最稳定
- 再大对真机清晰度收益很小，但包体会明显变大

### 2.2 动作主图

适用位置：
- Tab 2 动作卡片
- 动作详情页顶部 Hero
- Tab 1 训练步骤页当前动作图

推荐原图：
- `1200 x 1200`
- 比例：`1:1`

可接受下限：
- `960 x 960`

不建议超过：
- `1600 x 1600`

原因：
- 动作图在多个页面会被裁成不同大小
- 正方形素材最耐裁切
- 对动作主体居中的图片兼容最好

### 2.3 动作示例图

适用位置：
- Tab 2 动作详情页“动作示例”

推荐原图：
- `1400 x 1000`
- 比例：约 `7:5`

可接受下限：
- `1120 x 800`

不建议超过：
- `1800 x 1280`

原因：
- 当前示例图是横向卡片
- 比课程封面略紧凑，适合展示动作过程

## 3. 格式建议

优先级：

1. `JPEG`
2. `PNG`

使用原则：

- 照片类、真人拍摄、背景复杂：优先 `JPEG`
- 需要透明背景：才用 `PNG`
- 不要因为方便就所有图都用 `PNG`

原因：
- `PNG` 在照片场景下通常比 `JPEG` 大很多
- 你的 app 后续最容易把包体做大的就是图片和音频

## 4. 单张图片体积建议

建议控制：

- 课程封面：单张尽量 `200 KB - 500 KB`
- 动作主图：单张尽量 `150 KB - 400 KB`
- 动作示例图：单张尽量 `180 KB - 500 KB`

经验值：

- 单张超过 `1 MB`，通常就该先压缩再进项目
- 如果未来一次加很多图，几十张 `1 MB+` 图片会非常快地把 app 体积推高

## 5. 包体控制建议

如果你后面要补很多图，按这几个规则做：

1. 先裁尺寸，再压缩
不要把 `4000px` 原图直接丢进项目。

2. 一张图只保留一个面向当前 UI 的版本
不要同时保留“超大原图 + 实际展示图”。

3. 封面图和动作图分开处理
课程封面横图，动作图方图，不要混用。

4. 没透明需求就不用 `PNG`
这条对控制体积最有效。

5. 同一动作不要放太多示例图
如果一个动作放 5-8 张大图，体积会上升很快。
当前建议：
- 每个动作 `1-3` 张示例图就够

## 6. 命名建议

保持稳定的小写 kebab-case 或 snake_case。

当前项目已有命名风格：
- `course_strength_cover.png`
- `pushup_demo.png`
- `running_drill.png`

建议继续保持这种简单命名，不要带空格，不要带中文文件名。

## 7. 加图流程

建议流程：

1. 先在本地处理好尺寸
2. 导出压缩后的最终图片
3. 放到 `Resources/Images`
4. 拖进 Xcode 的 `Images` group
5. 确认 `AppFitMVP` target membership 已勾上
6. Build 真机检查

## 8. 真机检查清单

每次批量加图后，至少在真机上看这几项：

1. 课程封面裁切是否把主体切掉
2. 动作图主体是否居中
3. 动作示例图在详情页是否过糊
4. 列表滚动是否明显变卡
5. app 安装体积是否增长异常

## 9. 当前最稳的默认规则

如果你明天开始补图，不想想太多，直接按这套走：

- 课程封面：`1600 x 900`，`JPEG`
- 动作图：`1200 x 1200`，`JPEG`
- 动作示例图：`1400 x 1000`，`JPEG`
- 单张尽量不要超过 `500 KB`
- 有透明背景需求时再用 `PNG`

这样做，真机效果和包体通常都会比较平衡。
